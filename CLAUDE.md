# 점운 프로젝트 개발 헌법

이 문서는 Claude Code에게 주는 개발 헌법이다.
새 Claude Code 세션이 시작되면 이 문서를 처음부터 끝까지 읽고, 이 원칙을 따라 개발한다.
규칙을 모른 채 코드를 먼저 건드리는 것은 허용되지 않는다.

---

## 프로젝트 목표

대한민국 1위 AI 사주앱을 만드는 것이다.
사주아이의 장점을 벤치마킹하되 그대로 따라하지 않고,
AI 기술과 콘텐츠 설계를 결합하여 더 높은 완성도를 목표로 한다.

## 배포 현황 (실제 운영 중)

| 항목 | 내용 |
|------|------|
| 호스팅 | Vercel |
| 도메인 | jeomun.com |
| DB | Firebase Realtime Database |
| 결제 | Toss Payments |
| 파트너 | 서브도메인 방식 화이트라벨 |

로컬에서 코드를 수정하면 GitHub push 후 **1~2분 뒤** Vercel에 자동 배포된다.
즉시 반영이 아니므로 push 직후 바로 확인하지 말 것.

## 기술 스택

- **프레임워크**: Next.js (App Router, app/ 디렉터리 구조)
- **DB**: Firebase RTDB (`lib/firebase.ts`)
- **결제**: Toss Payments
- **스타일링**: 인라인 스타일 (Tailwind 미사용)
- **배포**: Vercel

> ⚠️ `lib/supabase.ts` 파일이 존재하지만 사용하지 않는다.
> Supabase 전환 계획이 있었으나 Firebase RTDB 유지로 결정이 났다.
> 이 파일을 활성화하거나 Supabase로 전환하는 코드를 절대 작성하지 말 것.

## 핵심 전략 — 점운은 실시간 AI가 아니다

```
만세력 계산
    ↓
규칙 엔진 (사주 원국, 십성, 신살, 대운, 세운)
    ↓
DB 템플릿 조회
    ↓
답변 출력
```

사용자는 AI와 대화한다고 느끼지만,
실제로는 API를 거의 사용하지 않는 초고속·저비용·안정적인 템플릿 기반 엔진으로 동작한다.

---

## 절대 원칙

1. **API 호출을 최소화한다.** 가능하면 API 없이 구현한다.
2. **템플릿으로 구현 가능한 것은 모두 템플릿으로 구현한다.**
3. 실시간 AI보다 속도·안정성·비용절감을 우선한다.
4. **기존 구조를 함부로 변경하지 않는다.** 항상 기존 코드를 이해한 후 수정한다.
5. 대규모 수정 전에는 반드시 이유를 설명한다.
6. **삭제보다 추가를 우선한다.**

---

## 콘텐츠 전략

AI는 **개발 단계에서만** 사용한다.
Claude를 이용해 사주 원국·십성·신살·대운·세운 등의 콘텐츠를 대량 생성하고,
생성된 콘텐츠는 DB에 저장한다.
서비스에서는 DB만 조회한다.

**목표**
- 질문·답변 DB: 5,000~10,000개 구축
- 답변 템플릿: 수만 개 구축
- API 없이 AI처럼 보이는 서비스 구현

## 현재 완성된 기능 (2026년 6월 기준)

### 사용자 앱 (`app/main-v2/`)
- 사주 원국 계산 (만세력, 일주·월주·연주·시주)
- 무료 결과: 오늘의 운세
- 유료 운세 (개별 구매 or 패키지):
  - 재물운, 연애운, 건강운, 성공운
  - 직업/사업운, 배우자운, 건강세부운
  - 부모/형제운, 자녀운
  - 올해운세(월별 포함), 30일 로드맵
- 패키지 상품: 기본분석(990원), 베이직·프리미엄·VIP
- 보관함: 구매 기록 저장 및 재열람 (`app/main-v2/history/`)
- 공유 기능: 카카오톡 등 링크 공유 (`app/main-v2/share/`)
- TTS 읽기 기능: 결과지·파트너결과지·공유·보관함 4곳
  - ↺ 처음부터 듣기 / 이어듣기 / 멈추기
  - 화면꺼짐 대비 및 자동 이어읽기 로직 적용

### 파트너 시스템 (`app/partner/`)
서브도메인 방식 화이트라벨 (1개 앱·DB·결제로 여러 파트너 운영)
- 파트너 브랜드 설정, 분석 생성, 결과지, 아카이브, 매출 관리
- 파트너 등급: free·basic·premium·diamond
- 다이아 파트너: 패키지 가격 직접 커스터마이징 가능
- 가입비 미입금 파트너 → 분석 생성 자체 차단

### 관리자 패널 (`app/admin/`)
- 고객 관리, 파트너 관리, 할인코드 관리, 매출 상위 고객

### 대운(大運) — 백엔드 완성, UI/결제 미완성
- 계산 함수 완성 (대운수·순행역행·60갑자·십이운성)
- 콘텐츠 완성 (십이운성 12단계 × 5섹션, 무료 미리보기/유료 전체해설)
- API 연동 완성 및 curl 검증 완료
- **미완성**: 타임라인 UI 화면, 결제 흐름

## Git 원칙

1. **중요 작업 전 반드시 Git 커밋부터 한다.** 작업 도중 문제가 생겨도 되돌아갈 수 있어야 한다.
2. 중요 작업 후 → `git commit` + `git push`
3. 항상 복구 가능하도록 유지한다.
4. 커밋 메시지는 한글로, 무엇을 왜 바꿨는지 명확히 적는다.
5. force push, reset --hard 등 되돌릴 수 없는 명령은 사용자 명시적 승인 없이 절대 실행하지 않는다.

## 개발 워크플로우

### 모든 작업에 공통 적용
1. 요청 내용을 파악한다
2. 관련 파일을 먼저 읽는다
3. 기존 코드 구조를 이해한 후 최소한만 수정한다
4. 빌드 오류 없는지 확인한다 (`npx tsc --noEmit`)
5. commit + push
6. 커밋 해시를 보고한다

### 큰 수정(여러 파일 수정 / 구조 변경 / 기능 추가)일 때
> **반드시 이 순서를 지킨다. 임의로 먼저 코드를 건드리지 않는다.**

1. **작업 전 Git 커밋** — 현재 상태를 먼저 저장한다
2. **계획을 먼저 설명한다** — 어떤 파일을 어떻게 바꿀 것인지 사용자에게 보고한다
3. **사용자 승인을 받는다** — "진행해", "해줘" 등 명시적 확인을 받은 후에만 코드를 수정한다
4. 수정 → 빌드 확인 → commit + push → 커밋 해시 보고

예외: 오타 수정, 텍스트 변경, 버튼 추가처럼 영향 범위가 명확히 작은 작업은 바로 진행해도 된다.
